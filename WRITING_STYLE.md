# WRITING_STYLE

새로운 문체를 제안하거나 출간본의 문장을 고치는 문서가 아닙니다. 22개 Chapter의 제목과 본문, CASE, Workflow, 실습, 마무리, 에필로그에서 실제로 반복되는 표현 방식을 정리합니다.

## 기본 서술자

### 확인된 규칙

- 본문의 중심 서술자는 1인칭 `저`다.
- 서술자는 완성된 방법을 밖에서 설명하기보다, 실제 업무를 겪고 실패하고 다시 살펴본 사람으로 등장한다.
- 경험 서술에서는 `저는`, `제가`, `제 머릿속` 같은 표현이 반복된다.
- 독자는 주로 실습과 마무리에서 `여러분` 또는 생략된 2인칭으로 호명된다.
- 저자와 독자의 경험을 섞어 `우리`라고 일반화하기보다, 먼저 저자의 장면을 보여준 뒤 독자에게 자신의 장면을 적도록 요청한다.

### 서술자의 위치

서술자는 다음 세 위치를 오간다.

1. **경험한 사람**  
   학교 현장에서 무엇을 보고, 누르고, 찾고, 멈췄는지 과거형으로 서술한다.

2. **뒤늦게 이해한 사람**  
   당시의 생각과 나중에 발견한 차이를 설명한다.

3. **함께 실습하는 안내자**  
   독자에게 한 번에 하나의 행동을 요청하고, 어디까지 하면 되는지 범위를 정한다.

## 기본 어조

### 확인된 규칙

- 전체적으로 존댓말을 사용한다.
- 경험 서술은 `-했습니다`, `-았습니다`, `-였습니다`의 과거형이 중심이다.
- 원칙과 해석은 `-합니다`, `-입니다`의 현재형으로 제시한다.
- 실습 지시는 `-합니다`, `-고릅니다`, `-적습니다`, `-확인합니다` 형식을 사용한다.
- 독자의 행동을 부드럽게 여는 문장에서는 `-해보세요`가 사용된다.
- 강압적인 명령형보다 수행 순서를 설명하는 안내형 문장이 중심이다.
- 독자의 실패나 미완성을 탓하지 않는다.
- 과장된 성공담이나 기술 낙관보다 작은 변화와 운영 가능성을 강조한다.

### 어조의 핵심 성격

- 회고적이다.
- 차분하다.
- 구체적이다.
- 판단을 서두르지 않는다.
- 실패를 숨기지 않는다.
- 독자를 평가하기보다 다음 행동으로 안내한다.
- AI의 능력보다 사람의 판단과 책임을 앞에 둔다.

## 시제 사용

| 문장 역할 | 주로 확인되는 시제 | 예시 형태 |
|---|---|---|
| 현장 장면 | 과거 | 열었습니다, 찾았습니다, 멈췄습니다 |
| 당시 생각 | 과거 | 생각했습니다, 알 수 없었습니다 |
| 뒤늦은 발견 | 과거 | 보였습니다, 알았습니다 |
| 책의 원칙 | 현재 | 남깁니다, 확인합니다, 필요합니다 |
| 실습 지시 | 현재 | 고릅니다, 적습니다, 펼칩니다 |
| 독자 제안 | 현재 청유·권유 | 떠올려보세요, 적어보세요 |
| 다음 단계 | 현재·미래 의미 | 다음에는 확인합니다, 이어갑니다 |

한 문단 안에서 시제를 임의로 섞기보다, 장면에서 해석으로 이동할 때 과거형에서 현재형으로 전환한다.

## 문장 길이와 리듬

### 짧은 문장

- 중요한 행동과 발견은 짧은 문장으로 분리한다.
- 한 문장만으로 문단을 만드는 방식이 반복된다.
- 장면 전환에도 짧은 문장을 사용한다.
- 예: `이름은 지웠습니다.`, `화면을 닫았습니다.`, `된다.`
- 짧은 문장은 장식용 구호보다 실제 행동, 멈춤 또는 인식 변화에 사용된다.

### 중간 길이 문장

- 대부분의 본문은 한두 가지 행동이나 생각을 담은 중간 길이 문장으로 구성된다.
- 하나의 문장에 여러 개념을 압축하기보다 문장을 나누어 순서를 보여준다.
- 접속어를 많이 겹치지 않고 앞 문장의 결과를 다음 문장이 이어받는다.

### 나열

- 반복 행동, 사람이 맡을 일, 확인 항목은 한 줄씩 나열한다.
- 같은 문장 구조를 반복해 리듬을 만든다.
- 쉼표로 긴 목록을 만드는 대신 줄을 나누는 경우가 많다.
- PART 마무리와 에필로그에서는 앞에서 수행한 행동을 짧은 문장으로 연속 나열한다.

## 문단 구성

### 확인된 규칙

- 문단은 비교적 짧다.
- 한 문단에는 한 장면, 한 판단 또는 한 변화만 둔다.
- 시간의 흐름이 바뀌거나 생각이 전환될 때 문단을 나눈다.
- 같은 표현을 다음 문단 첫머리에서 다시 받아 의미를 바꾸는 방식이 반복된다.
- 중요한 문장은 앞뒤에 여백을 두어 독립시킨다.
- 긴 설명 뒤에는 짧은 결론 문장을 둔다.
- 소제목 아래 첫 문장은 소제목을 다시 설명하기보다 다음 장면이나 판단을 바로 시작한다.

## Chapter 제목

### 확인된 경향

- Chapter 제목은 문장형이다.
- 다수의 제목이 1인칭 경험을 암시하는 과거형으로 끝난다.
- `만들기`, `설계 방법` 같은 설명형 명사 제목보다, 실제로 무엇을 겪거나 바꾼 문장을 사용한다.
- 성공뿐 아니라 실패와 멈춤도 제목에 포함한다.
- 제목은 해당 Chapter에서 일어난 인식 변화 또는 완성한 행동을 압축한다.
- 긴 제목은 의미 단위로 줄을 나누지만 문장 자체는 유지한다.

### 확인된 제목 유형

| 유형 | 출간본에서 확인되는 형태 |
|---|---|
| 문제 장면 | 하루 종일 바빴는데 아침 파일이 그대로 남음 |
| 실패 기록 | 화면부터 만들었음, 고칠수록 어긋남 |
| 행동 전환 | 화면을 닫고 업무를 적음 |
| 작은 완성 | 구조가 움직임, 업무가 끝까지 이어짐 |
| 원칙 확인 | 반복은 줄이고 판단은 남김 |
| 운영 결론 | 도구보다 오래 남는 것을 만듦 |

모든 Chapter 제목이 같은 문법으로 끝나는 것은 아니다. Chapter 3처럼 현재형 원칙 문장도 확인된다.

## 소제목

### 확인된 규칙

- 소제목도 대부분 문장형이다.
- 추상 명사만 두기보다 저자의 생각이나 행동 변화를 드러낸다.
- `처음에는 …했습니다`, `…인 줄 알았습니다`, `…을 나누었습니다` 같은 회고형 표현이 반복된다.
- 소제목은 이어지는 본문의 결론을 먼저 선언하기보다, 장면을 해석하는 방향을 제시한다.
- 질문형 소제목보다 서술형 소제목이 중심이다.

## 현장 장면 쓰기

### 시작 방식

- Chapter는 실제 행동이나 눈앞의 물건으로 시작하는 경우가 많다.
- 메신저 알림, 공문, 파일명, 입력창, 처치대, 표, 화면, 빈 종이처럼 구체적인 대상이 먼저 등장한다.
- `보건교사의 업무는 복잡합니다`처럼 일반론으로 시작하지 않는다.
- 시간, 화면, 손의 움직임과 업무 중단을 순서대로 보여준다.

### 장면에 포함되는 요소

- 무엇을 하려 했는가
- 중간에 무엇이 들어왔는가
- 무엇을 다시 찾거나 확인했는가
- 어디에서 손이나 생각이 멈췄는가
- 그때 무엇이라고 생각했는가

### 표현 방식

- 행동을 동사로 적는다: 열다, 찾다, 누르다, 지우다, 펼치다, 적다, 멈추다.
- 실제 화면이나 문서에 보이는 단어를 구체적으로 제시한다.
- 감정을 길게 해설하기보다 행동과 짧은 생각으로 드러낸다.
- 독자의 긴장감을 높이기 위한 과장된 묘사는 확인되지 않는다.

## 생각과 대화의 표기

### 직접 대화

- 실제 대화는 큰따옴표로 표시한다.
- 대화 앞뒤에 장면을 설명하는 문단을 둔다.
- 대화는 짧고 업무 상황을 드러내는 역할을 한다.
- 한 장면에 대화를 과도하게 이어가지 않는다.

### 속마음과 짧은 생각

- 당시의 속마음은 작은따옴표로 표시하는 경우가 있다.
- 매우 짧은 생각은 별도 문장으로 독립시키기도 한다.
- 속마음은 독자를 설득하기 위한 수사가 아니라, 당시의 오해나 멈춤을 보여준다.

## 실패를 서술하는 방식

### 확인된 규칙

- 실패한 결과를 먼저 숨기지 않고 구체적으로 적는다.
- 실패 원인을 개인의 능력 부족으로 단정하지 않는다.
- 당시에는 왜 괜찮다고 생각했는지 함께 보여준다.
- 문제를 발견한 흔적을 화면, 더미 데이터, 링크, 사용자 동선처럼 관찰 가능한 사실로 제시한다.
- 실패 뒤에는 즉시 거대한 해결책을 제시하지 않는다.
- 화면을 닫기, 종이를 꺼내기, 한 줄 적기처럼 다음의 작은 행동으로 이동한다.
- 실패는 성공담의 장식이 아니라 이후 구조를 바꾸는 근거로 사용된다.

## 개념을 설명하는 방식

### 구체적인 장면에서 개념으로 이동

개념을 먼저 정의한 뒤 사례를 붙이기보다 다음 순서가 반복된다.

```text
현장 장면
→ 당시의 생각
→ 반복되거나 어긋난 지점 발견
→ 두 가지 일을 구분
→ 짧은 원칙
→ 표·CASE·Workflow·실습으로 확인
```

### 대조

- `학생 방문`과 `링크를 다시 찾는 일`처럼 겉으로 함께 묶였던 일을 나눈다.
- `판단`과 `정리`, `화면`과 `순서`, `완성`과 `운영`을 대조한다.
- 한쪽을 무조건 제거하기보다 각각의 역할과 남겨야 할 자리를 밝힌다.
- 대조 결과는 핵심 정리, 핵심 메시지 또는 Before/After 표로 다시 압축한다.

### 반복되는 전환 표현

다음과 같은 전환 방식이 여러 Chapter에서 확인된다.

- 처음에는 …라고 생각했습니다.
- 하지만 …
- 그제야 …이 보였습니다.
- 이번에는 …
- 먼저 …
- 그 뒤에야 …
- 다시 펼쳤습니다.
- 하나만 남겼습니다.
- 아직 …하지 않아도 됩니다.

이 표현들은 동일한 문구를 모든 Chapter에 기계적으로 넣는 고정 서식은 아니다. 장면에서 발견과 실습으로 이동하는 공통적인 문장 기능으로 확인된다.

## 독자에게 지시하는 방식

### 한 번에 하나의 행동

- `최근 하루를 떠올려보세요`처럼 시작점을 좁힌다.
- 여러 업무가 아니라 장면 하나, 과제 하나, 반복 하나를 고르게 한다.
- 작성 전에 이전 Chapter의 결과물을 다시 펼치도록 안내한다.
- 실습 단계는 번호 또는 A·B·C 구분으로 나눈다.
- 각 단계에서 무엇을 적고 무엇을 아직 하지 않을지 함께 알려준다.

### 범위를 줄이는 문장

다음 기능의 문장이 반복된다.

- 잘 쓰려고 하지 않아도 된다고 안내한다.
- 모든 것을 바꾸지 않아도 된다고 말한다.
- 아직 해결 방법을 정하지 않아도 된다고 범위를 제한한다.
- 개인정보 없이 더미 데이터로 먼저 확인하도록 한다.
- 첫 시도는 한 장면·한 줄·한 행이면 충분하다고 안내한다.

### 확인 질문

- 질문은 독자의 판단을 대신하기보다 스스로 경계를 확인하게 한다.
- `-인가요?`, `-남아 있나요?`, `-할 수 있는가?` 형식을 사용한다.
- 실습 마지막에는 체크박스 질문으로 수행 여부를 확인한다.
- 질문 수는 작업을 진행할 수 있도록 목록으로 제한한다.

## CASE 문체

### 확인된 규칙

- 제목은 `CASE · 사례 제목` 형식이다.
- 실제 보건 업무의 특정 장면이나 한 건을 다룬다.
- 사례 안에서도 추상적인 성공 설명보다 사람이 한 일과 확인한 결과를 적는다.
- 사례는 본문에서 설명한 원칙을 새로운 이야기로 확장하기보다, 같은 원칙이 실제 업무에서 어떻게 보였는지 확인한다.
- 개인정보가 필요한 사례에서는 실명이나 식별정보를 쓰지 않는다.
- CASE의 마지막은 다음 Workflow 또는 실습으로 자연스럽게 이어진다.

## Workflow 문체

### 확인된 규칙

- 제목은 `Workflow · 흐름 제목` 형식이다.
- 긴 설명보다 짧은 행동 문장과 화살표로 순서를 제시한다.
- 각 단계는 명사보다 행동이 드러나는 표현을 사용한다.
- 순서 앞뒤에서 누가 판단하고 무엇을 확인하는지 본문으로 보완한다.
- Workflow를 자동으로 처리되는 기술 절차처럼만 쓰지 않는다.
- 사람이 판단하고 확인하는 단계가 문장에 남는다.

## 실습 문체

### 제목

- `실습 01 — 제목`부터 `실습 22 — 제목`까지 이어진다.
- 번호는 두 자리로 표기한다.
- 제목은 독자가 실제로 수행할 행동을 나타낸다.

### 본문

- 실습 시작에서 사용할 이전 결과물 또는 현재 장면을 밝힌다.
- 독자가 고를 대상의 수를 제한한다.
- 지시문은 짧고 순서대로 배치한다.
- 작성 예시와 독자 작성 영역을 구분한다.
- 민감정보를 적지 않는다는 안내가 필요한 실습에서는 입력 전에 경계를 먼저 제시한다.
- 실습 끝에서 아직 하지 않아도 되는 다음 단계를 분명히 하거나, 완성된 결과물을 확인한다.

## 핵심 정리와 핵심 메시지

- 본문에서 발견한 내용을 짧게 다시 묶는다.
- 새로운 주장을 추가하지 않는다.
- 두 범주를 나누거나 한 문장 원칙으로 압축한다.
- 본문의 설명보다 문장이 짧고 직접적이다.
- 사람의 판단, 책임, 개인정보 경계가 관련될 때는 단정적인 현재형을 사용한다.

## `쑤캥의 한마디`

### 확인된 규칙

- 1인칭 경험을 유지한다.
- 저자도 같은 지점에서 실패하거나 멈췄음을 밝힌다.
- 독자의 결과를 평가하지 않는다.
- 작은 시작을 인정하거나 다음 시도를 격려한다.
- 본문에서 확인되지 않은 새로운 기술 지식을 추가하지 않는다.
- Chapter의 정서적 마무리 역할을 한다.

## `오늘 바뀐 것`

### 문장 방식

- Before와 After를 한 문장씩 대조한다.
- Before는 처음의 오해, 큰 덩어리의 표현 또는 화면 중심의 생각을 담는다.
- After는 Chapter에서 실제로 바뀐 관점이나 구조를 담는다.
- 화살표로 변화 방향을 표시한다.
- After는 더 크고 화려한 결과보다 더 구체적인 관찰이나 운영 기준으로 바뀐다.

## Preview 문체

- 다음 Chapter에서 무엇을 만들지 모두 설명하지 않는다.
- 현재 Chapter의 마지막 결과를 다음 질문으로 연결한다.
- 짧은 설명 뒤에 한두 개의 질문을 두는 방식이 확인된다.
- 질문은 독자가 이미 만든 결과물을 다시 보게 한다.
- PART 마지막에서는 다음 PART Preview 또는 에필로그 Preview로 바뀐다.

## PART 마무리 문체

### 확인된 규칙

- 해당 PART에서 한 일을 과거형으로 되짚는다.
- Chapter별 핵심 행동을 짧은 문장으로 이어 붙인다.
- 성공만 정리하지 않고 남아 있는 질문과 다음 단계의 필요성을 적는다.
- 마지막 문장은 다음 PART의 시작 질문 또는 행동으로 연결된다.
- 긴 이론 요약보다 프로젝트가 실제로 어디까지 왔는지를 기록한다.

## 에필로그 문체

### 확인된 규칙

- Chapter 1의 장면과 표현을 다시 불러온다.
- 책 전체의 프로젝트 순서를 시간의 흐름으로 재서술한다.
- 앞에서 사용한 핵심 대조를 반복한다: 화면과 구조, 반복과 판단, 완성과 운영.
- 새로운 방법론을 추가하기보다 앞에서 확인한 의미를 다시 해석한다.
- 저자가 아직 운영하며 배우고 있다는 현재 상태를 밝힌다.
- 독자에게 책과 똑같은 결과물을 만들 필요가 없다고 말한다.
- 마지막에는 독자가 오늘 적을 수 있는 반복 한 줄로 돌아간다.

## 용어 표기

최종 조판본에서 다음 표기가 반복된다.

| 용어 | 확인된 표기 |
|---|---|
| 책의 대단위 | `PART` |
| 장 | `Chapter` |
| 인공지능 | `AI` |
| 업무 흐름 | `Workflow` |
| 사례 | `CASE` |
| 변화 비교 | `Before`, `After` |
| 프로젝트 진행 | `DAY`, `PART 진행률` |
| 실습 | `실습 01`처럼 두 자리 번호 |

- `PART`, `Chapter`, `Workflow`, `CASE`는 번역해 바꾸지 않고 혼용 표기를 유지한다.
- 책의 핵심 용어인 `Workflow`는 영문 첫 글자를 대문자로 쓴다.
- 일반 본문에서는 영어를 과도하게 늘리지 않고, 구조상 반복되는 라벨에 한정해 사용한다.
- 화면이나 파일명을 보여줄 때는 실제 보이는 표현을 그대로 적는다.

## 개인정보와 안전 문장

### 확인된 규칙

- 개인정보·민감정보 관련 문장은 모호하게 완화하지 않는다.
- 이름, 학번, 연락처, 질병명, 검사 결과, 상담 기록처럼 구체적인 정보 종류를 열거한다.
- 이름만 지우는 것으로 충분하지 않을 수 있음을 설명한다.
- 실제 자료보다 더미 데이터로 먼저 확인하도록 안내한다.
- AI가 판단할 수 있는 범위를 넓히기보다 사람이 판단하고 최종 확인할 범위를 먼저 적는다.
- 의학적 판단, 응급 대응, 상담, 공개 여부와 법적 책임은 사람에게 남긴다.
- 체크리스트와 경계 문장으로 독자가 다시 확인하게 한다.

## 피하는 표현 방식

최종 출간본에서 반복적으로 선택되지 않은 방식은 다음과 같다.

- 독자에게 능력 부족을 지적하는 표현
- AI가 모든 업무를 해결한다는 표현
- 한 번에 전체 업무를 자동화하라는 지시
- 근거 없이 `최고`, `완벽`, `혁신적`이라고 평가하는 표현
- 긴 이론 정의로 Chapter를 시작하는 방식
- 현장 맥락 없이 기능 이름만 나열하는 방식
- 실패를 삭제하고 완성된 결과만 제시하는 방식
- 개인정보를 실제 예시 데이터로 사용하는 방식
- 사람이 해야 할 판단을 AI의 결과로 대체하는 표현

이 목록은 금지어 사전이 아니라, 최종 출간본의 문체가 실제로 택한 방향과의 대비를 기록한 것이다.

## 일관성과 예외

### 반복해서 확인되는 일관성

- 1인칭 회고
- 존댓말
- 과거 장면에서 현재 원칙으로 이동
- 짧은 문단
- 구체적인 사물과 행동으로 시작
- 실패 또는 멈춤을 관찰 가능한 사실로 제시
- 한 번에 하나의 실습 행동
- 사람의 판단과 책임을 명시
- Chapter 결과를 다음 Chapter 질문으로 연결

### 확인된 예외

- 모든 Chapter가 새로운 현장 대화로 시작하지는 않는다. 이전 Chapter의 표나 Workflow를 다시 펼치는 것으로 시작하는 Chapter도 있다.
- 모든 소제목이 과거형은 아니다. 현재형 원칙이나 질문에 가까운 제목도 있다.
- 모든 본문에 직접 대화가 포함되지는 않는다.
- 에필로그는 실습 지시보다 회고와 독자 제안의 비중이 크다.
- CASE와 Workflow의 길이와 표현 형식은 다루는 업무에 따라 달라진다.
- PART 마무리는 PART별 Chapter 수와 남은 질문에 따라 분량이 다르다.

## 이 문서에서 확정하지 않는 항목

다음은 출간본에서 제작 기준으로 직접 확인할 수 없어 문체 규칙으로 확정하지 않는다.

- 문장을 생성한 특정 AI 모델
- 초고와 최종 원고 사이의 수정 횟수
- 저자가 의도한 문학적 장르명
- 모든 새 책에서 동일한 문장 수를 사용해야 한다는 기준
- 문단별 고정 글자 수
- 특정 표현을 자동으로 삽입하는 공식

Chapter 내부 요소의 정확한 순서와 필수·선택 항목은 `CHAPTER_TEMPLATE.md`에서 별도로 확인한다.
