# CHAPTER_TEMPLATE

## 문서 목적

이 문서는 『보건교사를 위한 AI 업무 자동화』의 22개 Chapter에서 반복되는 구성 요소와 실제 배열을 템플릿 형태로 기록합니다.

새로운 Chapter 형식을 만들거나 출간본의 원고를 수정하는 문서가 아닙니다. 모든 Chapter에 공통으로 확인되는 요소, 여러 Chapter에서 반복되는 선택 요소, PART 마지막 Chapter와 Chapter 1의 예외를 구분합니다.

## Chapter의 역할

출간본에서 Chapter는 정보를 설명하는 단위인 동시에 하나의 결과물을 만드는 프로젝트 단위다.

```text
현재 장면 또는 이전 결과물
→ 문제·멈춤·질문
→ 저자의 관찰과 해석
→ 핵심 원칙
→ CASE·Workflow로 확인
→ 독자 실습
→ 결과물 완성
→ Before/After 확인
→ 다음 Chapter·PART로 연결
```

각 Chapter의 프로젝트 진행 카드에는 `오늘 만드는 것`이 하나씩 적혀 있고, Chapter 후반의 `오늘 만든 프로젝트`에서 같은 결과물을 완료한다.

## 1. Chapter 표지

### 공통 구성

- 우측 상단의 큰 Chapter 번호
- `PART · CHAPTER` 라벨
- 문장형 Chapter 제목
- 제목 아래 구분선
- 하단 책 제목
- 페이지 번호

### 확인된 규칙

- Chapter마다 본문과 분리된 독립 표지를 사용한다.
- 표지는 네이비 전면 배경이다.
- Chapter 제목은 해당 Chapter에서 실제로 일어난 변화나 발견을 문장으로 적는다.
- 제목이 길면 의미 단위로 줄을 나눈다.
- 제목만 읽어도 책 전체 프로젝트의 진행을 따라갈 수 있다.

시각 수치는 `BOOK_STYLE.md`에 별도로 기록되어 있다.

## 2. 프로젝트 진행 카드

### 공통 필드

```text
프로젝트 진행 카드
DAY [숫자] · PART [번호] · Chapter [PART 안 순번] / [PART의 Chapter 수]
PART 진행 [상태] · [앞에서 완료한 수] / [PART의 Chapter 수]
걸리는 시간 약 [숫자]분
난이도 [★ 표시]
오늘 만드는 것 [결과물]
```

### 확인된 상태

- PART 첫 Chapter: `PART 진행 시작 · 0 / n`
- PART의 다음 Chapter: `PART 진행 진행 중 · x / n`

`PART 진행 진행 중`처럼 `진행`이 반복되는 표현도 최종 출간본에 그대로 확인된다. 이 문서는 이를 교정하지 않는다.

### 결과물 일치

진행 카드의 `오늘 만드는 것`은 Chapter 후반의 `오늘 만든 프로젝트`에서 같은 결과물로 확인된다.

예:

```text
오늘 만드는 것 업무 설계표
→
오늘 만든 프로젝트
✓ 업무 설계표
```

## 3. Chapter 도입

### 유형 A: 현장 장면 시작

주로 앞부분 Chapter에서 확인된다.

- 메신저 알림
- 공문과 파일
- 입력창
- 처치대
- 온라인 보건실 화면
- Google Sheets 셀
- 실제 공유 순간

### 유형 B: 이전 결과물 다시 펼치기

프로젝트가 누적된 뒤의 Chapter에서 반복된다.

- `Chapter n에서 만든 …을 다시 펼쳤습니다.`
- `PART n에서 만든 …을 다시 열었습니다.`
- 이전 표와 새 종이를 나란히 놓는다.
- 두 결과물을 함께 펼치고 연결한다.

### 도입의 공통 기능

1. 지금 어디에서 시작하는지 보여준다.
2. 손이 실제로 하는 행동을 적는다.
3. 멈춘 자리 또는 남은 질문을 드러낸다.
4. 이번 Chapter의 결과물이 왜 필요한지 만든다.

일반적인 이론 정의나 기능 소개로 시작하는 Chapter는 확인되지 않는다.

## 4. 본문 전개

### 반복 순서

```text
처음 생각했던 것
→ 직접 해본 행동
→ 예상과 달랐던 결과
→ 멈춘 지점
→ 두 가지 일을 구분
→ 이번 Chapter의 원칙
```

### 소제목

- 문장형으로 적는다.
- 저자의 오해, 멈춤, 발견 또는 행동 변화를 담는다.
- 본문을 주제별로 분리하지만 별도의 번호 체계를 항상 사용하지는 않는다.
- 모든 Chapter의 소제목 수가 같지는 않다.

## 5. 핵심 정리와 핵심 메시지

### 출현

- Chapter 1–2: `핵심 정리`
- Chapter 3–22: `핵심 메시지`

### 기능

- 앞 본문의 발견을 짧게 다시 묶는다.
- 새로운 개념을 추가하지 않는다.
- 두 범주를 나누거나 한 문장 원칙으로 압축한다.
- 실습에서 확인할 기준을 독자에게 넘긴다.

### 배치

대체로 본문 해석 뒤, CASE·Workflow 또는 실습 전에 배치된다. CASE와의 앞뒤 순서는 Chapter에 따라 달라진다.

## 6. CASE

### 출현

- Chapter 3–22
- Chapter 1–2에는 CASE 라벨이 없다.

### 기본 형태

```markdown
### CASE · [구체적인 사례 제목]

[업무 한 건의 상황]
[사람이 실제로 한 행동]
[확인된 결과]
```

### 기능

- 본문 원칙을 실제 보건 업무 한 건에 놓아본다.
- 추상적인 개념을 행동·표·화면·기록으로 확인한다.
- 다음 Workflow 또는 실습의 재료를 제공한다.
- 개인정보가 필요한 실제 식별정보는 사용하지 않는다.

CASE의 세부 작성 방식은 `CASE_GUIDE.md`에서 별도로 기록한다.

## 7. Workflow

### 출현

Workflow 라벨은 다음 Chapter에서 확인된다.

- Chapter 4
- Chapter 6–7
- Chapter 10–22

Chapter 1–3, 5, 8–9에는 Workflow 라벨이 없다.

### 기능

- CASE 또는 본문에서 확인한 행동을 순서로 바꾼다.
- 시작과 끝, 다음 행동을 보여준다.
- 실습 전에 독자가 따라갈 흐름을 제시한다.
- 사람이 판단하는 단계가 사라지지 않는다.

### 배치 변형

- CASE 앞에 Workflow가 놓이는 Chapter가 있다.
- CASE 뒤에 Workflow가 놓이는 Chapter가 있다.
- CASE와 Workflow가 같은 페이지에 이어지는 Chapter가 있다.

따라서 `CASE 다음에는 항상 Workflow`라는 고정 순서는 출간본에서 확인되지 않는다.

## 8. 실습

### 공통 제목

```text
실습 01 — [행동 제목]
...
실습 22 — [행동 제목]
```

- 실습 번호는 책 전체에서 연속된다.
- 번호는 두 자리로 표기한다.
- Chapter 번호와 실습 번호가 일치한다.
- 제목은 독자가 실제로 수행할 행동을 나타낸다.

### 실습 시작

- 앞 Chapter 또는 PART의 결과물을 다시 펼친다.
- 현재 Chapter에서 고를 장면이나 업무의 수를 제한한다.
- 아직 하지 않아도 되는 일을 먼저 밝히기도 한다.
- 개인정보가 필요한 경우 실제 자료 대신 더미 데이터 경계를 제시한다.

### 실습 본문

실습에 따라 다음 요소가 조합된다.

- 번호 단계
- A·B·C 구분
- 작성 예시
- 독자 작성 표
- 빈칸 문장
- 화살표 흐름
- 체크박스
- 마지막 완성 문장

모든 실습이 동일한 단계 수나 동일한 표 형식을 사용하지 않는다.

### 실습 끝

- 결과를 다시 읽거나 손가락으로 따라가도록 한다.
- 체크리스트로 완료 여부를 확인한다.
- 이번 단계에서 멈출 범위를 밝힌다.
- 완성된 결과물을 `오늘 만든 프로젝트`로 넘긴다.

## 9. 주의

### 라벨이 확인되는 Chapter

- Chapter 2
- Chapter 3
- Chapter 4
- Chapter 6

### 내용

- 실제 개인정보 대신 더미 데이터를 사용한다.
- 이름만 지웠다고 안전한 것은 아님을 알린다.
- 아직 원인을 단정하지 않는다.
- 여러 증상을 하나의 오류로 묶지 않는다.
- 해결책을 너무 일찍 쓰지 않는다.

다른 Chapter에도 개인정보·안전 확인 문장이 있지만 모두 `주의` 상자를 사용하지는 않는다. 따라서 주의 상자는 내용상 필요할 때 나타나는 선택 요소다.

## 10. 작은 축하

### 출현

```markdown
### 작은 축하

[이번 실습에서 실제로 완성된 작은 변화]
```

## 11. 오늘 만든 프로젝트

### 공통 구성

- `오늘 만든 프로젝트`
- 체크 표시와 결과물명
- PART 진행률
- 일부 Chapter의 전체 프로젝트 진행률

### 결과물명

결과물명은 진행 카드의 `오늘 만드는 것`과 대응한다.

### PART 마지막 Chapter

PART의 마지막 결과 영역은 `n / n 완료`를 표시한다.

확인 예:

- PART 1 · 2 / 2 완료
- PART 5 진행률 · 3 / 3 완료
- PART 8 진행률 · 3 / 3 완료

표현은 완전히 하나로 통일되어 있지 않다.

## 12. 자료실 QR

### 출현

- Chapter 1: 없음
- Chapter 2–22: 있음

### 기본 안내

- 책의 자료실이 열린다는 설명
- 해당 PART 실습 자료를 내려받아 작성하라는 안내
- 프로젝트 파일을 다음 PART에서도 계속 사용한다는 안내

### 위치

대체로 `오늘 만든 프로젝트` 뒤와 `쑤캥의 한마디` 앞에 배치된다.

Chapter 1은 같은 PART의 첫 결과만 만든 뒤 QR 없이 Chapter 2로 이동한다. PART 1을 완료하는 Chapter 2에서 처음 QR이 나온다.

세부 규칙은 `QR_POLICY.md`에 별도로 기록한다.

## 13. 쑤캥의 한마디

### 출현

- 22 / 22 Chapter

### 기능

- 저자도 같은 장면에서 멈췄거나 오해했음을 밝힌다.
- 이번 Chapter의 결과를 저자의 경험으로 다시 확인한다.
- 독자의 작은 시작을 인정한다.
- 다음 행동을 재촉하지 않는다.
- 새로운 이론이나 기능을 추가하지 않는다.

### 위치

대체로 `오늘 만든 프로젝트`와 QR 뒤, `오늘 바뀐 것` 앞에 놓인다.

## 14. 오늘 바뀐 것

### 출현

- 22 / 22 Chapter

### 기본 형태

```markdown
## 오늘 바뀐 것

| Before |  | After |
|---|---|---|
| [이전 생각·상태] | ↓ | [Chapter 뒤의 변화] |
```

### 기능

- Chapter 시작 전의 해석과 실습 뒤의 해석을 비교한다.
- 결과물이 무엇인지 다시 나열하기보다 관점이나 운영 방식의 변화를 보여준다.
- After는 더 크고 화려한 결과보다 더 구체적인 기준이나 행동을 담는다.

## 15. Preview

### 공통 기능

- 현재 만든 결과를 확인한다.
- 다음 단계에서 다룰 질문을 남긴다.
- 현재 결과만으로 부족한 지점을 보여준다.
- 다음 결과를 미리 완성해서 설명하지 않는다.

### 유형 A: 다음 Chapter Preview

PART 안에서 다음 Chapter로 이동할 때 사용한다.

```markdown
### 다음 Chapter Preview

[이번 Chapter에서 완성한 것]
[다음 Chapter에서 확인할 것]
[다음 단계가 필요한 질문]
```

### 유형 B: 다음 PART Preview

명시적 라벨은 Chapter 4, 7, 10의 PART 마무리에서 확인된다.

```markdown
### 다음 PART Preview

[이번 PART의 최종 결과]
[다음 PART에서 시작할 행동]
[남은 질문]
```

### 유형 C: 에필로그 Preview

Chapter 22와 PART 8 마무리 뒤에 사용된다.

```markdown
### 에필로그 Preview

[책 전체 프로젝트에서 남은 마지막 이야기]
```

### PART 5–7 마지막 Chapter

Chapter 13, 16, 19에는 `다음 Chapter Preview`가 먼저 배치되고, 이어지는 `PART n을 마치며`가 다음 PART로 연결된다. 라벨명만 보면 다음 Chapter이지만 실제 위치는 PART의 마지막 Chapter다. 이 문서는 최종 출간본의 표기를 그대로 기록한다.

## PART 안 위치에 따른 차이

### PART 첫 Chapter

- 진행 카드가 `0 / n`에서 시작한다.
- PART의 입력 결과물 또는 새 현장 장면을 제시한다.
- PART 목표를 바로 완성하지 않고 첫 중간 결과물을 만든다.

### PART 중간 Chapter

- 앞 Chapter 결과물을 다시 펼친다.
- 첫 결과의 모호함, 중복 또는 부족한 기준을 찾는다.
- PART 최종 결과에 필요한 두 번째 재료를 만든다.

### PART 마지막 Chapter

- 앞의 두 결과물을 함께 펼치거나 연결한다.
- PART의 최종 결과물을 완성한다.
- PART 진행률을 `n / n 완료`로 표시한다.
- Preview 또는 PART 마무리를 통해 다음 PART로 넘긴다.

PART 1–2는 Chapter가 2개이므로 `중간 Chapter`가 없다.

## Chapter 작성 상태 확인표

### 표지와 진행 카드

- [ ] Chapter 표지가 본문과 독립되어 있다.
- [ ] Chapter 번호와 PART·CHAPTER 라벨이 있다.
- [ ] 제목이 이번 Chapter의 실제 변화와 일치한다.
- [ ] DAY, PART, Chapter 순번이 실제 구조와 맞는다.
- [ ] 예상 시간과 난이도가 표시되어 있다.
- [ ] `오늘 만드는 것`이 하나의 결과물로 적혀 있다.

### 도입과 본문

- [ ] 실제 현장 장면 또는 이전 결과물에서 시작한다.
- [ ] 이번 Chapter에서 멈춘 지점이나 질문이 보인다.
- [ ] 처음 생각과 실제 확인 결과가 구분된다.
- [ ] 소제목이 본문의 행동·인식 변화와 연결된다.
- [ ] 핵심 정리·메시지가 본문에 없던 새 주장을 추가하지 않는다.

### CASE·Workflow

- [ ] CASE를 사용하는 경우 실제 업무 한 건과 연결된다.
- [ ] Workflow를 사용하는 경우 행동 순서가 보인다.
- [ ] 사람이 판단하고 확인할 단계가 남아 있다.
- [ ] CASE와 Workflow를 필요 없이 모두 삽입하지 않는다.
- [ ] 개인정보 또는 민감정보가 실제 식별 가능한 형태로 들어가지 않는다.

### 실습

- [ ] 실습 번호가 Chapter 번호와 일치한다.
- [ ] 실습 제목이 독자의 실제 행동을 나타낸다.
- [ ] 사용할 이전 결과물 또는 시작점이 분명하다.
- [ ] 한 번에 고를 장면·업무의 범위가 제한되어 있다.
- [ ] 작성 예시와 독자 작성 영역이 구분된다.
- [ ] 실습 마지막에 확인 질문 또는 완성 문장이 있다.
- [ ] 아직 하지 않아도 되는 다음 단계가 필요한 경우 분명히 적혀 있다.

### 결과와 연결

- [ ] `오늘 만든 프로젝트` 결과물이 진행 카드와 일치한다.
- [ ] PART 진행률의 분자·분모가 실제 위치와 맞는다.
- [ ] Chapter 2–22에는 자료실 QR 안내가 있다.
- [ ] `쑤캥의 한마디`가 새로운 내용을 추가하지 않는다.
- [ ] Before/After가 이번 Chapter의 실제 변화를 비교한다.
- [ ] Preview가 현재 결과에서 다음 단계로 이어진다.
- [ ] PART 마지막 Chapter라면 PART 마무리와의 연결을 확인한다.

이 확인표는 최종 출간본에서 확인되는 Chapter 요소를 대조하기 위한 것이다. 새로운 구성 요소를 추가하는 허가 목록이 아니다.

## 확인된 예외

- Chapter 1에는 CASE, Workflow, QR, 작은 축하가 없다.
- Chapter 2에는 CASE와 Workflow가 없지만 QR이 처음 등장한다.
- CASE는 Chapter 3부터 모든 Chapter에서 확인된다.
- Workflow는 16개 Chapter에만 있다.
- `주의` 라벨 상자는 4개 Chapter에만 있다.
- 작은 축하는 Chapter 7부터 등장한다.
- Chapter 4, 7, 10은 `다음 PART Preview`로 연결된다.
- Chapter 13, 16, 19는 PART의 마지막 Chapter지만 `다음 Chapter Preview` 라벨을 사용한 뒤 PART 마무리로 이어진다.
- Chapter 22는 다음 Chapter 대신 에필로그로 연결된다.
- 표·체크리스트·빈칸 문장의 수와 형식은 실습마다 다르다.
- CASE와 Workflow의 앞뒤 순서는 고정되어 있지 않다.
- PART 마무리와 Chapter 결과 영역이 같은 페이지에 시작되는 경우와 다음 페이지로 이어지는 경우가 있다.

## 이 문서에서 확정하지 않는 항목

다음 내용은 출간본에서 새로운 책의 고정 규칙으로 확인되지 않는다.

- 모든 새 책이 반드시 22개 Chapter를 가져야 한다는 기준
- 모든 Chapter가 반드시 같은 페이지 수여야 한다는 기준
- 모든 Chapter에 CASE와 Workflow를 모두 넣어야 한다는 기준
- 모든 실습이 같은 단계 수와 같은 표를 사용해야 한다는 기준
- 모든 Chapter에 주의 상자를 넣어야 한다는 기준
- 모든 Chapter 제목을 과거형으로 써야 한다는 기준
- 모든 Chapter에 QR을 하나씩 배치해야 한다는 기준
- 각 컴포넌트의 문장 수를 고정하는 기준

프로젝트북 전체의 결과 누적과 독자 수행 규칙은 `PROJECTBOOK_RULES.md`에서 별도로 기록한다.
