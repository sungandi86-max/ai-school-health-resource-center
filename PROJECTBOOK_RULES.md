# PROJECTBOOK_RULES

## 문서 목적

이 문서는 『보건교사를 위한 AI 업무 자동화』가 독자가 읽기만 하는 설명서가 아니라, 하나의 프로젝트를 단계별로 수행하고 기록하는 프로젝트북으로 작동하는 규칙을 정리합니다.

새로운 교육 설계 원칙을 제안하는 문서가 아닙니다. 최종 출간본의 8개 PART, 22개 Chapter, 22개 실습과 결과물, DAY·진행률, QR 자료 연결에서 반복 확인되는 작동 방식을 기록합니다.

## 확인 범위

- 프로젝트 진행 카드 22개
- 실습 01–22
- 오늘 만든 프로젝트 22개
- PART 진행률과 전체 프로젝트 진행률
- Chapter·PART 사이의 이전 결과물 재사용
- 사람·AI·정보 경계
- 실패 기록과 재설계
- 자료실 QR과 프로젝트 파일 연속 사용 안내
- DAY 1–90
- 실제 학교 운영과 에필로그

## 프로젝트북의 기본 정의

출간본에서 프로젝트북은 다음과 같이 작동한다.

```text
실제 업무 장면을 관찰한다
→ 한 번에 하나의 결과물을 만든다
→ 그 결과물을 다음 Chapter에서 다시 사용한다
→ 실패와 확인 결과를 기록한다
→ 사람의 판단과 정보 경계를 남긴다
→ 실제 업무에서 사용한다
→ 멈춘 자리를 기록하고 고친다
→ 다른 사람이 이어갈 운영 기준을 남긴다
```

책의 프로젝트는 화면이나 자동화 결과만을 최종 산출물로 삼지 않는다. 관찰표, 경계표, 실패 지도, 설계표, Workflow, 운영 기록과 선언문이 모두 프로젝트 결과물이다.

## 1. 실제 업무 장면에서 시작한다

### 확인된 규칙

- 프로젝트는 도구나 기능 선택으로 시작하지 않는다.
- Chapter 1은 메신저, 공문, 학생 방문과 다시 찾는 행동에서 시작한다.
- 이후 Chapter도 입력창, 화면, 표, 셀, 공유 순간처럼 실제 업무에서 보이는 장면을 출발점으로 사용한다.
- 프로젝트가 누적된 뒤에는 이전 결과물을 실제 책상 위에 다시 펼치는 장면으로 시작한다.
- 추상적인 문제명보다 사람이 무엇을 했고 어디에서 멈췄는지를 먼저 적는다.

### 프로젝트북에서의 기능

독자가 자신의 업무를 일반적인 `검진 업무`, `자료 관리`로 적는 데서 멈추지 않고, 실제로 다시 찾고 다시 쓰고 다시 확인한 행동을 발견하게 한다.

## 2. 한 Chapter에서 결과물 하나를 만든다

### 확인된 규칙

22개 프로젝트 진행 카드에는 각각 `오늘 만드는 것`이 하나씩 적혀 있다.

Chapter 후반의 `오늘 만든 프로젝트`는 같은 결과물명을 체크 표시와 함께 다시 제시한다.

```text
오늘 만드는 것 [결과물]
→
실습 [번호]
→
오늘 만든 프로젝트
✓ [같은 결과물]
```

## 3. 결과물을 누적한다

### 확인된 규칙

- Chapter 2는 Chapter 1의 하루 한 장면 해부표를 다시 펼친다.
- Chapter 4는 Chapter 3의 경계표와 PART 1의 반복 장면 기록지를 함께 사용한다.
- Chapter 7은 Chapter 5–6의 실패 기록을 한곳에 펼친다.
- Chapter 10은 Chapter 8–9의 행동과 기준을 연결한다.
- Chapter 13은 Chapter 11–12의 행·열 설계를 실제 구조에 옮긴다.
- Chapter 16은 Chapter 14–15의 Workflow 초안과 중복 제거 결과를 결합한다.
- Chapter 19는 PART 6의 Workflow와 Chapter 18의 AI 역할표를 함께 펼친다.
- Chapter 22는 PART 1부터 만든 결과물을 한곳에 펼친다.

### 누적 방식

```text
앞 결과물을 보관한다
→ 다음 Chapter에서 다시 펼친다
→ 부족한 기준을 추가한다
→ 두 결과물을 연결한다
→ 새 결과물로 확인한다
```

앞 결과물을 폐기하고 매 Chapter 새 파일로 다시 시작하는 방식은 확인되지 않는다.

## 4. 범위를 하나로 줄인다

### 반복되는 범위 단위

- 하루 전체가 아니라 한 장면
- 모든 반복이 아니라 세 장면
- 세 장면 전체가 아니라 첫 개선 과제 하나
- 큰 업무가 아니라 실제 행동 한 줄
- 한 행에는 업무 한 건
- 모든 자동화가 아니라 반복 한 줄
- 전체 시스템 재구축이 아니라 사용자가 멈춘 작은 지점

### 확인되는 안내 표현

- 하나만 고릅니다.
- 한 장면이면 충분합니다.
- 한 줄만 남깁니다.
- 이번 달의 한 가지를 적습니다.
- 업무 한 건을 고릅니다.
- 반복 하나부터 시작합니다.

범위를 줄이는 것은 내용을 단순화하는 장식 문구가 아니다. 각 실습의 입력 수와 결과물 범위를 실제로 제한한다.

## 5. 해결보다 관찰이 먼저다

### PART 1

- 아직 해결 방법을 적지 않는다.
- 무엇을 바꿀지 바로 고르지 않는다.
- 중단 뒤 되풀이한 행동을 관찰한다.

### PART 2

- 무엇을 자동화할지 결정하기 전에 사람이 판단할 일과 정보 경계를 정한다.
- 모든 반복을 바꾸지 않고 첫 과제 하나를 고른다.

### PART 3

- 오류를 즉시 고치기 전에 직접 본 결과와 아직 확인하지 않은 것을 나눈다.
- 원인을 미리 단정하지 않고 사용자가 한 행동과 실제 결과를 기록한다.

이 순서 때문에 프로젝트북의 앞부분 결과물은 완성 화면이 아니라 관찰표, 경계표와 실패 기록이다.

## 6. 아직 하지 않을 일을 명시한다

### 확인된 문장 기능

각 단계는 무엇을 할지뿐 아니라 지금은 무엇을 하지 않을지도 밝힌다.

확인 예:

- 아직 해결 방법을 적지 않는다.
- 아직 무엇을 먼저 바꿀지 정하지 않는다.
- 아직 요청 문장을 쓰지 않는다.
- 아직 표의 구조를 만들지 않는다.
- 원인을 적지 않는다.
- 다시 만들지 않는다.
- 프로그램을 만들지 않는다.
- 화면을 붙이지 않는다.
- 자동화하지 않는다.

### 프로젝트북에서의 기능

- 앞 단계를 건너뛰지 않게 한다.
- 독자가 한 실습에서 너무 많은 결과를 만들지 않게 한다.
- 확인되지 않은 원인을 해결책처럼 적지 않게 한다.
- 화면과 AI가 필요한 시점을 뒤로 미룬다.

`하지 않을 일`의 내용은 Chapter마다 다르며, 모든 Chapter에 같은 문장으로 들어가지는 않는다.

## 7. 실패를 프로젝트 결과로 남긴다

### 확인된 구조

PART 3은 첫 온라인 보건실의 실패를 세 개의 결과물로 기록한다.

```text
첫 이상 징후 기록
→ 사용자 흐름 실패 지도
→ 재설계 기준 메모
```

### 확인된 규칙

- 실패 화면을 성공 화면으로 바로 교체하지 않는다.
- 보이는 결과와 직접 확인한 결과를 구분한다.
- 아직 확인하지 않은 항목을 남긴다.
- 사용자가 한 행동, 기대한 결과, 실제 결과를 나란히 적는다.
- 서로 다른 증상을 한 화면의 버그로 묶지 않는다.
- 실패에서 빠진 질문을 찾아 다음 설계의 기준으로 바꾼다.

따라서 이 출간본에서 실패 기록은 미완성 원고가 아니라 정식 프로젝트 단계다.

### 사람에게 남기는 일

출간본에서 반복 확인되는 항목:

- 의학적 판단
- 응급 대응
- 상담
- 실제 조치
- 연락 대상과 시점 결정

- 최종 승인과 사용

- AI 입력·공개 금지

실제 프로젝트에서는 더미 데이터로 먼저 확인하고, 이름 하나만 지우는 것으로 충분한지 다시 살핀다.

## 9. 설명을 실습으로 바꾼다

### 공통 순서

```text
저자의 장면
→ 문제 해석
→ 핵심 정리·메시지
→ CASE·Workflow
→ 독자 실습
```

### 확인된 규칙

- Chapter마다 실습이 하나 있다.
- Chapter 번호와 실습 번호가 일치한다.
- 실습 제목은 독자가 수행할 행동을 나타낸다.
- 실습은 앞 본문에서 설명하지 않은 새 과제를 갑자기 추가하지 않는다.
- 앞에서 본 사례와 기준을 독자의 업무에 적용하게 한다.

## 10. 독자가 책 안에서 직접 작성하게 한다

### 실습에서 확인되는 작성 방식

- 빈 표
- 예시가 채워진 표
- 독자 작성용 표
- 밑줄 또는 빈칸 문장
- 체크박스
- A·B·C 구분
- 번호 단계
- Before/After
- 역할표
- 흐름도
- 선언문

### 예시와 작성 영역

예시가 있는 실습은 저자의 업무 예와 독자의 빈 영역을 구분한다.

예:

```text
작성 예시
→
나의 하루 한 장면
```

독자가 답을 머릿속으로만 생각하게 하지 않고 표, 한 줄, 체크 또는 문장으로 흔적을 남기게 한다.

## 11. 실습의 완료를 다시 확인한다

### 확인 수단

- 마지막 확인 질문
- 체크박스
- 완성 문장
- 손가락으로 처음부터 끝까지 따라가기
- 소리 내어 읽기
- 원문과 결과 비교
- 다른 계정·기기·사용자 동선 확인
- `오늘 만든 프로젝트` 체크 표시

### 완료의 의미

완료는 단순히 빈칸을 채운 상태가 아니다.

Chapter에 따라 다음을 확인한다.

- 한 행동에 시작과 끝이 있는가
- 한 행에 업무 한 건만 있는가
- 같은 열이 같은 뜻으로 쓰이는가
- 멈춘 자리에서 다음 행동을 찾을 수 있는가
- 사람이 판단할 단계가 남아 있는가
- 개인정보가 불필요하게 들어가지 않았는가
- 실제 사용자가 같은 동선을 따라갈 수 있는가

## 12. 작은 성공을 표시한다

### 출현

`작은 축하`는 Chapter 7–22에서 확인된다.

### 확인된 기능

- 실패가 다음 설계 기준이 된 순간을 결과로 인정한다.
- 큰 업무 안에서 실제 행동이 보이기 시작한 것을 확인한다.
- 업무 구조가 같은 기준 안에서 움직인 것을 확인한다.
- 첫 Workflow가 끝까지 이어진 것을 확인한다.
- 반복 한 줄을 안전하게 맡긴 것을 확인한다.
- 운영 기록과 운영 선언문이 완성된 것을 확인한다.

작은 축하는 자동화나 화면의 규모가 아니라, 이번 Chapter에서 실제로 확인된 변화에 붙는다.

## 14. Before/After로 인식 변화를 기록한다

### 출현

- Chapter 1–22 전체

### 확인된 규칙

Before는 Chapter 시작 전의 큰 해석이나 오해를 적는다.

After는 실습 뒤에 생긴 구체적인 관찰, 기준 또는 운영 방식으로 바뀐다.

예시 기능:

```text
일이 많아서 바쁘다
→
중단될 때마다 같은 일을 다시 시작해서 바쁘다
```

```text
AI가 업무를 대신한다
→
AI는 반복을 돕고 사람은 판단을 이어간다
```

Before/After는 결과물 이름을 복사하는 표가 아니라, Chapter에서 바뀐 관점을 한 문장으로 남기는 장치다.

## 15. 결과를 다음 질문으로 넘긴다

### 확인된 규칙

- Chapter 끝에는 다음 단계 Preview가 있다.
- 현재 결과가 무엇인지 확인한 뒤, 그 결과만으로 해결되지 않은 질문을 남긴다.
- 다음 Chapter는 그 질문을 새로운 결과물로 바꾼다.
- PART 마지막에서는 다음 PART 또는 에필로그로 연결된다.

### 연결 예

```text
반복 장면 기록지
→ 무엇부터 바꾸고 무엇은 사람이 판단해야 하는가
→ 사람·AI·정보 경계표
```

```text
업무 설계표
→ 실제 구조에서 같은 기준으로 움직이는가
→ 업무 단위·컬럼 설계·첫 업무 구조
```

```text
AI Workflow 1.0
→ 다른 사람도 같은 기준으로 사용할 수 있는가
→ 운영 기록표·운영 체크리스트
```

## 16. 책과 자료실의 프로젝트 파일을 이어 쓴다

### 확인된 규칙

- Chapter 2–22에 자료실 QR이 있다.
- QR은 해당 PART의 실습 자료를 내려받도록 안내한다.
- `이 프로젝트 파일은 다음 PART에서도 계속 이어서 사용합니다`라는 안내가 반복된다.
- 독자는 각 PART에서 독립된 완성 파일을 새로 만드는 것이 아니라 같은 프로젝트 기록을 이어간다.
- 책의 실습과 자료실의 파일은 같은 PART 구조를 공유한다.

Chapter 1에는 QR이 없고, PART 1을 완료하는 Chapter 2에서 처음 자료실을 연결한다.

## 17. 실제 사용 전에는 더미 데이터로 확인한다

### 확인된 규칙

- 학생 이름, 학번, 연락처, 질병명, 검사 결과와 상담 기록을 넣지 않는다.
- 이름을 지워도 학년·반·날짜·드문 상황의 조합으로 식별될 수 있음을 확인한다.
- 실제 자료를 옮기기 전에 개인정보가 없는 더미 데이터로 시험한다.
- 다른 계정, 공개 범위와 권한을 확인한다.
- 잘못되었을 때 중단하거나 되돌릴 수 있는지 확인한다.
- 최종 확인자를 정한다.

더미 데이터는 실습용 장식이 아니라 실제 사용 전 안전 확인 단계로 작동한다.

## 18. 화면보다 구조를 먼저 만든다

### 확인된 순서

```text
업무 분해 초안
→ 행동 기준표
→ 업무 설계표
→ 업무 단위 정의표
→ 컬럼 설계표
→ 첫 업무 구조
→ Workflow
```

### 확인된 규칙

- 화면 이름보다 실제 행동을 적는다.
- 한 줄에는 행동 하나만 둔다.
- 행동의 시작, 끝과 다음 행동을 적는다.
- 한 행에는 시작과 끝이 있는 업무 한 건을 둔다.
- 열마다 현재 필요한 이유를 확인한다.
- 같은 열에는 같은 기준을 사용한다.
- 아직 화면이나 자동화를 붙이지 않은 상태에서도 구조를 결과물로 인정한다.

## 19. AI는 Workflow가 만들어진 뒤 연결한다

### 확인된 순서

- PART 6에서 첫 Workflow를 완성한다.
- PART 7에서 첫 AI 요청문을 만든다.
- 사람과 AI의 역할을 다시 나눈다.
- 기존 Workflow 한곳에 작은 AI 역할을 연결한다.

### 확인된 규칙

- 새로운 AI Workflow를 처음부터 다시 만들지 않는다.
- 기존 Workflow의 반복되는 한 부분을 표시한다.
- 사람이 이미 판단한 내용만 정리하도록 요청한다.
- 결과를 원문과 비교한다.
- 공개 전에 사람이 최종 확인한다.
- 판단, 공개와 완료는 사람에게 남긴다.

## 20. 실제 운영을 프로젝트에 포함한다

### Chapter 20

- 실제 학교에서 공유한다.
- 첫 사용자가 한 행동을 기록한다.
- 기대한 흐름과 실제 흐름을 비교한다.
- 사용자가 멈춘 지점을 적는다.
- 가장 작은 부분을 수정한다.
- 같은 동선으로 다시 확인한다.
- 수정 전후를 운영 기록표에 남긴다.

### 확인된 원칙

프로젝트의 첫 운영 결과는 `완벽하다`는 확인이 아니다. 실제 사용자가 멈춘 자리를 발견하고 기록한 것이 결과다.

## 21. 다른 사람이 이어갈 기준을 남긴다

### Chapter 21

- 운영 역할을 나눈다.
- 할 수 있는 일과 하면 안 되는 일을 적는다.
- 최종 확인자를 남긴다.
- 수정 전과 수정 후의 확인 순서를 적는다.
- 마지막 확인일과 수정 이유를 기록한다.

### Chapter 22

- 처음 줄이고 싶었던 반복
- 사람에게 끝까지 남길 판단
- 맡길 반복의 범위
- 넣지 않을 정보
- 운영하며 확인할 순서
- 다음 사람이 이어갈 기록

을 운영 선언문으로 정리한다.

프로젝트는 만든 사람이 기억하는 도구에서, 다른 사람이 같은 기준으로 이어갈 수 있는 운영으로 이동한다.

## 22. 완료 뒤에도 운영을 남긴다

### 확인된 규칙

- PART 8에서 전체 프로젝트 진행률은 8 / 8이 된다.
- PART 8 마무리는 운영이 완료 표시로 끝나지 않는다고 적는다.
- 에필로그는 Workflow와 온라인 보건실이 지금도 바뀌고 있다고 밝힌다.
- 학교 일정과 사용자가 달라지면 다시 사용자의 자리에서 따라가고, 멈춘 지점을 기록하고, 작게 고치고, 수정 이유를 남긴다.

따라서 `완료`는 더 이상 수정하지 않는 최종 상태가 아니라, 운영을 시작할 수 있는 기준이 남은 상태로 확인된다.

## 실습의 기본 작동 구조

```text
1. 사용할 이전 결과물 또는 실제 장면을 정한다.
2. 한 장면·한 업무·한 행동으로 범위를 줄인다.
3. 작성 예시를 확인한다.
4. 독자 자신의 표·빈칸·흐름을 작성한다.
5. 개인정보와 사람의 판단 경계를 확인한다.
6. 체크리스트 또는 완성 문장으로 다시 검토한다.
7. 오늘 만든 프로젝트에 결과물명을 남긴다.
8. 다음 Preview에서 남은 질문을 확인한다.
```

모든 실습이 정확히 8개 번호 단계로 구성되는 것은 아니다. 위 구조는 22개 실습에서 반복되는 기능의 순서다.

| 관찰 | 실제 장면과 반복 행동이 기록됨 |

| 운영 선언 | 도구가 바뀌어도 지킬 기준이 문장으로 남음 |

## 이 문서에서 확정하지 않는 항목

다음 내용은 출간본에서 새로운 프로젝트북의 고정 규칙으로 확인되지 않는다.

- 모든 프로젝트북이 반드시 8개 PART와 22개 실습을 가져야 한다는 기준
- 모든 프로젝트가 반드시 90일이어야 한다는 기준
- 모든 Chapter의 실습 시간이 같아야 한다는 기준
- 모든 결과물이 표 형식이어야 한다는 기준
- 모든 Chapter에 CASE와 Workflow가 있어야 한다는 기준
- 모든 프로젝트가 AI나 자동화를 최종 목표로 해야 한다는 기준
- 모든 새 책이 동일한 보건 업무 사례를 사용해야 한다는 기준
- 모든 PART에서 전체 프로젝트 진행률을 표시해야 한다는 기준

이 문서는 이 출간본에서 확인되는 프로젝트북 작동 규칙을 기록한다. 사례, Workflow, 자료와 QR의 세부 규칙은 각각의 후속 문서에서 별도로 정리한다.
